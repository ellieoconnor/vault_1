## Key Concepts Covered

### 1. Test Design Workflow
Plan tests BEFORE writing them — design before code.

Steps:
1. Load Context — understand the feature/system
2. Risk/Testability Assessment — score probability × impact
3. Coverage Planning — determine what to test and at which level
4. Generate Test Design Document — blueprint for implementation

Use when: new features, major refactors, quality gate before development.

### 2. Risk & Testability Assessment
**Risk = Probability × Impact**

| Factor | Low | Medium | High |
|--------|-----|--------|------|
| Probability | Simple, stable | Moderate complexity | Complex, frequent changes |
| Impact | Aesthetic | User pain | Business-critical |

Examples:
- Login: Probability High × Impact High = P0
- Tooltip: Probability Low × Impact Low = P3

Testability considerations:
- Can we test in isolation?
- Are there blocking dependencies?
- Do we need infrastructure first?

Testability issues often signal tight coupling — let it inform design.

### 3. Coverage Planning: Test Levels
- **Unit** — isolated, fast, no external deps (pure functions, business logic)
- **Integration** — multiple components (DB, API, service layers)
- **E2E** — full user workflows in real browser (critical paths, user journeys)

Coverage strategy:
| Priority | Unit | Integration | E2E | Manual |
|----------|------|-------------|-----|--------|
| P0 | ✅ | ✅ | ✅ | ✅ |
| P1 | ✅ | ✅ | ✅ | — |
| P2 | — | ✅ | — | ✅ |
| P3 | — | — | — | ✅ |

### 4. Test Priorities Matrix
Coverage targets: 100% P0 · 80% P1 · 50% P2 · 20% P3

Real project example (password reset feature):
- `POST /auth/forgot-password` → P0 → all levels
- Reset link expiry → P0 → all levels
- UI loading spinner → P3 → manual or skip
