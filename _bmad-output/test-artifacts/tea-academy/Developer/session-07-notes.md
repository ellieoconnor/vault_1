### fixture-architecture.md — Composable Fixture Patterns

**Core Principle:** Build test helpers as pure functions first, then wrap in framework-specific fixtures. Compose with `mergeTests`, never inheritance.

**Why not Page Object Model inheritance:**
- Inheritance chains (BasePage → LoginPage → AdminPage) create tight coupling
- Changes to base break all descendants
- Cannot mix-and-match capabilities freely

**Pure Function → Fixture Pattern:**
```typescript
// Step 1: Pure function — no framework dependency, unit-testable
export async function apiRequest({ request, method, url, data }: ApiRequestParams) {
  const response = await request.fetch(url, { method, data });
  if (!response.ok()) throw new Error(`Failed: ${response.status()}`);
  return response.json();
}

// Step 2: Fixture wrapper — inject framework at boundary
export const test = base.extend<{ apiRequest: typeof apiRequest }>({
  apiRequest: async ({ request }, use) => {
    await use((params) => apiRequest({ request, ...params }));
  }
});
```

**Composing with mergeTests:**
```typescript
// One concern per fixture, compose freely
export const test = mergeTests(base, apiRequestFixture, networkFixture, authFixture);
```

**Auto-cleanup pattern:**
```typescript
seedUser: async ({}, use) => {
  const createdUsers: string[] = [];
  const seedUser = async (userData) => {
    const user = await seedDatabase('users', userData);
    createdUsers.push(user.id);
    return user;
  };
  await use(seedUser);
  // Auto-cleanup after test
  for (const userId of createdUsers) await deleteRecord('users', userId);
}
```

**Rule of thumb:**
- 3+ uses → fixture with subpath export
- 2-3 uses → utility module
- 1 use → keep inline (avoid premature abstraction)

## Key Takeaways

- Pure functions first — testable without framework overhead
- Inject framework dependencies at the fixture boundary only
- `mergeTests` > inheritance — compose freely, no coupling
- Fixtures own their cleanup — no scattered `afterEach` in test files
