## Key Concepts Covered

### 1. Testing as Engineering
- Tests are designed before written (like architecture before coding)
- Tests have quality standards — not just "does it run?"
- Tests are maintained like production code
- Testing decisions are risk-based

**Key Principle:** If you wouldn't accept sloppy production code, don't accept sloppy test code.

### 2. Risk-Based Testing: P0-P3 Matrix
- **P0 — Critical (Must Test):** Login, payment, data loss, security — business fails if broken
- **P1 — High (Should Test):** Core user workflows, key features, data integrity — major user pain
- **P2 — Medium (Nice to Test):** Secondary features, edge cases with workarounds — inconvenience
- **P3 — Low (Optional):** Tooltips, help text, aesthetic issues — minimal impact

**Formula:** Risk = Probability × Impact

### 3. TEA Definition of Done
- No flaky tests (deterministic pass/fail)
- No hard waits/sleeps (use `waitFor` conditions)
- Stateless & parallelizable (no shared state between tests)
- No order dependency (every block works in isolation)
- Self-cleaning tests (delete/deactivate entities they create)
- Tests live near source code (co-located)
- Low maintenance (avoid brittle selectors, use APIs for state setup)

## Key Takeaways

- Testing is engineering: design before write, maintain like production code
- Risk-based testing: prioritize P0/P1, be deliberate about P2/P3
- Definition of Done: deterministic, stateless, self-cleaning, low maintenance
- Quality standards = engineering rigor, not perfectionism
- Builds trust in CI/CD by preventing flaky, brittle test suites
