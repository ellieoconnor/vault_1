## What Went Well

1. **100% story delivery** — all 6 stories shipped, deployed, and secure
2. **Story 1.2 established durable patterns** — session config, Argon2, Zod validation middleware, central error handler; every subsequent story reused these without friction
3. **SEC-001 properly closed** — session fixation vulnerability identified during 1.4, scoped to 1.6, and resolved with `session.regenerate()` pattern
4. **Teaching approach landed well** — Developer is learning Prisma and React simultaneously; the story note format supported learning alongside delivery
5. **"Previous Story Learnings" sections** — warnings about NodeNext `.js` imports, session patterns, etc. were useful reference material across stories

---

## Challenges & Growth Areas

1. **Stories explained the "what" but not the "why"** — tasks read like instructions without context. Developer didn't have a clear sense of the bigger picture or end goal for each story. This made every task feel arbitrary and made troubleshooting harder.
2. **Critical warnings buried in Dev Notes** — the NodeNext `.js` import extension rule, `credentials: 'include'` requirement, and Prisma v7 quirks appeared as CRITICAL warnings in 4–5 consecutive stories. A learner scanning the Tasks section wouldn't see them until hitting the problem.
3. **Flow between stories felt disconnected** — no narrative connecting what was just built to what comes next and why it matters.
4. **Zero frontend tests across 6 stories** — all React code shipped untested. Not a one-story oversight — a structural gap heading into Epic 2.
5. **Architecture doc path mismatch** — `client/server` (in doc) vs `apps/frontend/apps/backend` (actual) repeated in every feature story. The doc needs updating, not just per-story warnings.

---

## Key Insights & Learnings

1. **The "why" gap is the most important fix for Epic 2.** When Developer doesn't understand the purpose of a task, good judgment calls become impossible. Stories need to open with context, not tasks.
2. **Test-first on `zoneCalculator.ts`** is the highest-value testing investment for Epic 2. Pure function, known inputs, known outputs — and the entire progress bar system depends on it being correct.
3. **Tailwind CSS v4 + shadcn/ui are pre-Epic 2 requirements.** These weren't set up in Epic 1, but every Epic 2 UI story needs them. They're prep work, not story work.
4. **Developer's instinct to flag the zone logic before building it** is exactly the right product thinking. That kind of early question-raising should be encouraged and rewarded throughout Epic 2.
5. **`removeQueries` vs `invalidateQueries`** — a non-obvious TanStack Query distinction discovered in Story 1.5 that should be documented for future stories involving auth state changes.
